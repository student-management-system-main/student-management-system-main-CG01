# Requirements Document

## Introduction

The School/Institution Fee Management System is a web-based platform that enables educational institutions to manage student fee records, process payments, generate invoices, and proactively identify at-risk accounts using an AI-powered risk scoring engine. The system provides administrators with a centralized dashboard, automated notifications for overdue fees, and exportable reports. The AI/ML module uses Logistic Regression and Decision Tree models (scikit-learn) to score each student's likelihood of defaulting on fee payments, enabling early intervention.

## Glossary

- **System**: The School/Institution Fee Management System as a whole
- **Admin**: An authorized institution staff member who manages fee records, students, and reports
- **Student**: An enrolled individual whose fee records are tracked by the System
- **Fee**: A monetary obligation assigned to a Student for a specific period or service
- **Invoice**: A formal document generated by the System itemizing fees owed by a Student
- **Transaction**: A recorded payment event linked to a Student and one or more Invoices
- **Risk_Score**: A numerical value (0–100) computed by the AI_Module representing the probability that a Student will default on fee payment
- **Risk_Category**: A classification derived from the Risk_Score: Low (0–39), Medium (40–69), High (70–100)
- **AI_Module**: The Python-based machine learning component that computes Risk_Scores using Logistic Regression and Decision Tree models
- **Notification_Engine**: The backend service responsible for sending email and SMS reminders and escalation alerts
- **API**: The RESTful backend layer (Flask/Django) that exposes endpoints for fee management, authentication, and notifications
- **Dashboard**: The React/HTML-based admin interface for viewing and managing fee data
- **Report**: A structured summary of fee collection, outstanding balances, or risk data that can be exported
- **Escalation**: An automated workflow triggered when a Student's overdue balance or Risk_Category exceeds a defined threshold

## Requirements

### Requirement 1: Student Fee Record Management

**User Story:** As an Admin, I want to create, view, update, and deactivate student fee records, so that I can maintain accurate and up-to-date fee obligations for every enrolled student.

#### Acceptance Criteria

1. THE API SHALL expose endpoints to create, read, update, and deactivate Student fee records stored in the MySQL database.
2. WHEN an Admin submits a new Student fee record with all required fields, THE API SHALL persist the record and return a unique Student identifier within 2 seconds.
3. WHEN an Admin submits a Student fee record with missing required fields, THE API SHALL return a descriptive validation error specifying each missing field.
4. WHEN an Admin updates a Student fee record, THE System SHALL log the previous values, the new values, the Admin's identifier, and the timestamp of the change in the Logs table.
5. WHILE a Student record is deactivated, THE Dashboard SHALL display the record as inactive and prevent new Invoices from being assigned to it.

---

### Requirement 2: Fee and Invoice Management

**User Story:** As an Admin, I want to define fee structures and generate invoices for students, so that students receive accurate billing for their obligations.

#### Acceptance Criteria

1. THE API SHALL allow Admins to define fee types (e.g., tuition, library, lab) with an amount, currency, and due date.
2. WHEN an Admin triggers invoice generation for a Student, THE System SHALL create an Invoice record linking the Student, the applicable fee types, the total amount due, and the due date.
3. WHEN an Invoice is created, THE System SHALL assign it a unique Invoice identifier and set its initial status to "Unpaid".
4. WHEN an Invoice's due date passes and its status remains "Unpaid", THE System SHALL automatically update the Invoice status to "Overdue".
5. THE API SHALL allow Admins to apply partial or full payments to an Invoice, updating the outstanding balance and status accordingly.
6. WHEN an Invoice's outstanding balance reaches zero, THE System SHALL update the Invoice status to "Paid" and record the completion timestamp.
7. IF an Admin attempts to generate a duplicate Invoice for the same Student, fee type, and billing period, THEN THE API SHALL reject the request and return an error indicating the duplicate.

---

### Requirement 3: Payment Transaction Recording

**User Story:** As an Admin, I want to record and view all payment transactions, so that I have a complete and auditable history of all fee payments.

#### Acceptance Criteria

1. WHEN a payment is submitted, THE API SHALL create a Transaction record containing the Student identifier, Invoice identifier, amount paid, payment method, and timestamp.
2. THE API SHALL return the created Transaction record, including its unique Transaction identifier, within 2 seconds of submission.
3. IF a submitted payment amount is less than or equal to zero, THEN THE API SHALL reject the Transaction and return a validation error.
4. IF a submitted payment amount exceeds the outstanding Invoice balance, THEN THE API SHALL reject the Transaction and return an error specifying the maximum allowable amount.
5. THE System SHALL maintain an immutable Transaction log; once recorded, a Transaction SHALL NOT be deleted or modified, only reversed via a separate reversal Transaction.
6. WHEN a reversal Transaction is created, THE System SHALL restore the Invoice's outstanding balance and update the Invoice status accordingly.

---

### Requirement 4: AI-Powered Risk Scoring

**User Story:** As an Admin, I want each student to be assigned an AI-computed risk score, so that I can identify students likely to default on payments and intervene early.

#### Acceptance Criteria

1. THE AI_Module SHALL compute a Risk_Score between 0 and 100 for each Student using a trained Logistic Regression model and a trained Decision Tree model, then return the higher of the two scores as the final Risk_Score.
2. WHEN a Transaction is recorded or an Invoice status changes, THE System SHALL trigger a Risk_Score recomputation for the affected Student within 5 minutes.
3. THE AI_Module SHALL derive Risk_Scores from features including payment history, number of overdue Invoices, total outstanding balance, enrollment duration, and historical default rate.
4. WHEN a Risk_Score is computed, THE System SHALL persist the score, the Risk_Category, the model version used, and the computation timestamp in the Risk_Scores table.
5. WHEN a Student's Risk_Category changes from a lower to a higher category, THE System SHALL trigger the Notification_Engine to alert the assigned Admin.
6. THE Dashboard SHALL display each Student's current Risk_Score and Risk_Category with a visual indicator (color-coded: green for Low, amber for Medium, red for High).
7. THE AI_Module SHALL expose a batch scoring endpoint that recomputes Risk_Scores for all Students on a configurable schedule (default: daily at midnight).
8. WHERE model retraining is enabled, THE AI_Module SHALL retrain models on the latest Transaction and Invoice data and replace the active model only when the new model's cross-validated accuracy exceeds the current model's accuracy.

---

### Requirement 5: Notification and Escalation Engine

**User Story:** As an Admin, I want the system to automatically send reminders and escalation alerts to students and staff, so that overdue fees are addressed promptly without manual follow-up.

#### Acceptance Criteria

1. WHEN an Invoice transitions to "Overdue" status, THE Notification_Engine SHALL send an email reminder to the Student's registered email address within 15 minutes.
2. WHERE SMS notifications are enabled for a Student, THE Notification_Engine SHALL also send an SMS reminder to the Student's registered phone number when an Invoice transitions to "Overdue".
3. WHEN an Invoice remains "Overdue" for 7 days without a payment, THE Notification_Engine SHALL send a second reminder to the Student and a notification to the assigned Admin.
4. WHEN an Invoice remains "Overdue" for 30 days without a payment, THE Notification_Engine SHALL trigger an Escalation workflow, notifying senior Admin staff via email.
5. IF a notification delivery fails, THEN THE Notification_Engine SHALL retry delivery up to 3 times with exponential backoff before logging the failure in the Logs table.
6. THE System SHALL record every notification attempt in the Logs table, including the recipient, channel (email/SMS), timestamp, and delivery status.
7. WHEN a Student's outstanding balance is fully paid, THE Notification_Engine SHALL suppress any pending scheduled reminders for that Student's resolved Invoices.

---

### Requirement 6: Admin Dashboard and Payment Records View

**User Story:** As an Admin, I want a centralized dashboard that shows fee collection status, outstanding balances, and risk indicators, so that I can monitor the institution's financial health at a glance.

#### Acceptance Criteria

1. THE Dashboard SHALL display a summary panel showing total fees collected, total outstanding balance, and count of Overdue Invoices for the current billing period, refreshed at most every 60 seconds.
2. THE Dashboard SHALL provide a searchable and filterable table of Students showing name, total outstanding balance, number of Overdue Invoices, and Risk_Category.
3. WHEN an Admin selects a Student from the table, THE Dashboard SHALL display the Student's full payment history, all Invoices with their statuses, and the current Risk_Score.
4. THE Dashboard SHALL allow Admins to filter the Student table by Risk_Category, Invoice status, and billing period.
5. WHILE data is loading, THE Dashboard SHALL display a loading indicator and remain interactive for other operations.
6. IF the API returns an error response, THEN THE Dashboard SHALL display a user-readable error message without exposing internal error details or stack traces.

---

### Requirement 7: Reports and Data Export

**User Story:** As an Admin, I want to generate and export reports on fee collection and risk data, so that I can share financial summaries with institutional leadership.

#### Acceptance Criteria

1. THE System SHALL allow Admins to generate Reports filtered by date range, Student group, fee type, Invoice status, and Risk_Category.
2. WHEN an Admin requests a Report, THE API SHALL return the Report data within 10 seconds for datasets up to 10,000 records.
3. THE System SHALL support exporting Reports in CSV and PDF formats.
4. WHEN an Admin exports a Report, THE System SHALL include a generation timestamp, the applied filters, and the Admin's identifier in the exported file.
5. THE System SHALL provide a pre-built "High Risk Students" Report listing all Students with a Risk_Category of High, their outstanding balances, and their last payment date.
6. WHEN a Report export is initiated, THE System SHALL log the export event including the Admin identifier, Report type, format, and timestamp.

---

### Requirement 8: Authentication and Authorization

**User Story:** As an institution, I want role-based access control for all system functions, so that only authorized staff can access or modify sensitive fee data.

#### Acceptance Criteria

1. THE API SHALL authenticate all requests using JSON Web Tokens (JWT); unauthenticated requests SHALL receive an HTTP 401 response.
2. THE System SHALL support at least two roles: "Admin" with full read/write access and "Viewer" with read-only access to Dashboard and Reports.
3. WHEN an authenticated user attempts an action outside their role's permissions, THE API SHALL return an HTTP 403 response with a descriptive error message.
4. WHEN an Admin logs in with valid credentials, THE API SHALL issue a JWT with an expiry of 8 hours and a refresh token with an expiry of 7 days.
5. IF a JWT is expired or invalid, THEN THE API SHALL return an HTTP 401 response and THE Dashboard SHALL redirect the user to the login page.
6. THE System SHALL enforce HTTPS for all API and Dashboard communications.
7. THE API SHALL hash all passwords using bcrypt with a minimum cost factor of 12 before storing them in the database.

---

### Requirement 9: Audit Logging

**User Story:** As an institution, I want a complete audit trail of all system actions, so that I can investigate discrepancies and demonstrate compliance.

#### Acceptance Criteria

1. THE System SHALL record an audit log entry for every create, update, deactivate, and export action performed through the API, including the actor's identifier, the affected resource type and identifier, the action performed, and the UTC timestamp.
2. THE System SHALL retain audit log entries for a minimum of 2 years.
3. WHEN an Admin queries the audit log, THE API SHALL return results filtered by date range, actor, resource type, or action type within 5 seconds.
4. THE System SHALL store audit log entries in an append-only Logs table; existing log entries SHALL NOT be modifiable or deletable through any API endpoint.

---

### Requirement 10: System Availability and Performance

**User Story:** As an institution, I want the system to be reliable and responsive under normal operating conditions, so that fee management operations are not disrupted.

#### Acceptance Criteria

1. THE API SHALL respond to fee record read requests within 500 milliseconds at the 95th percentile under a load of up to 100 concurrent users.
2. THE API SHALL respond to write requests (create/update Transaction, Invoice) within 1 second at the 95th percentile under a load of up to 100 concurrent users.
3. THE System SHALL achieve an uptime of at least 99.5% measured over any rolling 30-day period, excluding scheduled maintenance windows.
4. WHEN a scheduled maintenance window is planned, THE System SHALL display a maintenance notice on the Dashboard at least 24 hours in advance.
5. IF the database connection is unavailable, THEN THE API SHALL return an HTTP 503 response and log the connectivity failure with a timestamp.
